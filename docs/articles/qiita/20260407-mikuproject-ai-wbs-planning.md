## はじめに

計画を立てて実行するときに、やるべきことを整理し、全体像を見えるようにする方法のひとつがWBSです。

私はソフトウェア開発にまつわる業界にいますが、WBS作成はとても大切な作業だと感じています。その一方で、まあ面倒な作業でもあります（個人の感想です）。

慣れてくると、叩き台そのものは比較的すぐに作れるようになります。それでも手間がかかることに変わりはありませんし、立案した内容を電子的なファイルなどに落とし込む作業もまた面倒です。さらに、作成の前提や分解のしかたを誤ると、その後の計画や実行に影響が出ます。場合によっては、金銭的にも具合の悪い結果につながります。

他方、WBS作成には定番のツールもあります。経験者の方であれば、良くも悪くも `Microsoft Project` に代表されるような製品を思い浮かべる人も多いでしょう。

ただ、今回ここで掘り下げたいのは、個別の製品の良し悪しではありません。現場で実際に使われている道具や運用が、生成AIとの往復を前提にした流れへ、どこまで自然につながるのか、という点です。

定番ツールがあるからといって、それを導入すれば現場がうまく回る、というほど単純でもありません。現実には、`Excel` ベースの愛用WBS様式で運用されたり、組織独自の仕組みやソフトで回されたり、あるいは OSS のツールが使われたりと、さまざまな形でWBSが取り回されています。

さて、そんななか、私たちの仕事に生成AIを使うことがあたりまえの時代になってきました。そうなると、当然ながらWBS作成でも生成AIを最大限に活用したくなります。

ただ、素の生成AIとの対話だけでWBS作成を完結させるのは、現時点では少し不便です。生成AIでWBSの叩き台を作ることはできても、それを見える形にしたり、実務で扱いやすい形へ落とし込んだりするには、やはり何らかの補助的なツールがほしくなります。

私は、生成AIを前提にする時代なら、計画策定を支えるツールの構造自体も、生成AIとうまく協働できる形へ変わるべきではないかと考えています。人が最初から最後まで手で入力するだけではなく、まず生成AIにWBSの叩き台を作らせ、その結果を人が見直し、整え、必要に応じてまたAIとやり取りする。そうした往復を自然に扱えることが、これからは重要になるはずです。

そこで今回は、生成AIと一緒にWBSの叩き台を作り、その結果を `mikuproject` で `WBS Excel ブック (.xlsx)` や `Markdown`、`SVG` などの形に落とし込むところまでの流れを見ていきます。生成した結果をさらに磨いたり、既存計画へ反映したりする話は、また別の記事で扱う予定です。

## 背景

### mikuscore の開発プロセス

mikuscore では、強い意味での AI駆動開発を採用しています。

* ソースコードは人間が直接編集しない
* ソースコードの変更はすべて生成AIが行う
* 仕様は Markdown（md）ファイルに記述する
* mdファイルの編集も原則として生成AIに任せる
* 人間による手動編集は最小限に抑える

  * 理由：人間の介入は一貫性を崩し、その後のAI編集を困難にするため

=> このため、mdファイルは単なる仕様書ではなく、
**AIが作業するためのインターフェース**として機能しています。

### 現在の技術課題（ABC譜面）

現在、ABC記譜法の読み書き機能を強化しています。その中で、以下の問題が発生しています。

* ABC読み込み処理で不具合が多発している
* 不具合は個別ではなく、同種のものが繰り返し発生している
* 原因を分析すると、正規表現に起因するケースが多い
* さらに、生成AIのトークン上限に達すると、作業が完全に停止する

  * （いわゆる「MPが足りない」状態になる）

### 気づき

ABC読み込み処理は、もはや単純な文字列処理ではなく、

=> **構文を持つ「言語処理」になっている**

と考えられます。

そのため：

* 正規表現ベースの処理には限界がある
* 字句解析（lexer）＋構文解析（parser）への移行が必要

### 方針

以上を踏まえ、本計画では以下をおこないます。

* ABC読み込み処理を、正規表現ベースから
  **字句解析（lexer）＋構文解析（parser）へ移行する**
* 上記の実現のために、
  **生成AIの制約（特にトークン上限）を前提としたWBSを設計する**

## 条件と制約

### 開発時間の制約

* 作業は土日祝日、または平日の夜間に限定される

  * （実作業の多くは生成AIが担当するが、人間の関与も一定必要）

### AI駆動開発の制約

* トークン上限に到達すると、作業が完全に停止する

  * 思考が浅くなるのではなく、「何もできなくなる」
* Rate limit や Premium request 上限も存在する
* 生成AIは非常に高速だが、連続実行には制約がある

### 技術的前提

* 多数の小さなテストが既に存在しており、挙動変化は検知可能
* 本改修は UI を伴わない
* 仕様が比較的予測しやすい

=> そのため、実装から単体テストまでは
**生成AIがほぼ自律的に実行可能**

### 作業分担

* 実装・単体テスト：生成AIが主体
* 統合テスト：人間（2〜3日間程度必要）
* git/GitHub コミット・プッシュ・マージ・リリース：人間
* リリース後の記事執筆（Qiita / note）：人間(もちろん生成AI併用)

### AI作業準備の制約

* 生成AIに作業を依頼する前に、必要なmdファイルを事前に準備する必要がある
* これは実装そのものとは別の工程として扱う必要がある
* mdが不十分な状態で実装を開始すると、生成AIの挙動が不安定になる

=> そのため、本プロジェクトでは以下の順序を前提とする：

1. md（仕様・制約・タスク）の整備
2. 生成AIによる実装・テスト
3. 人間による最終確認

=> mdの準備は「補助作業」ではなく、
**AI駆動開発における主要工程のひとつ**である

## WBS設計の方針

本プロジェクトでは、通常のWBSとは異なる設計が必要になります。

=> 本プロジェクトでは、コードよりも先にmdを設計する。

理由は明確で、

=> **生成AIのMP（トークン上限）が枯渇すると、作業が完全停止するため**

です。

そのため、WBSは「効率」ではなく、

=> **「停止しないこと」**

を最優先に設計します。

### 設計原則

* 1タスクは小さく、単独で完結すること
* タスクは途中で停止しても成果が残ること
* 重いタスクは事前に分解すること
* タスク間の依存関係を最小化すること

## `mikuproject-ai-json-spec` とは何か

`mikuproject-ai-json-spec` は、生成AIと `mikuproject` の間で project 情報をどう受け渡すかを整理した仕様です。

ポイントは、生成AIとのやり取りを `MS Project XML` ではなく `JSON` ベースで行うところにあります。しかも、何でも全量を渡すのではなく、用途に応じた `projection JSON` を使い分ける方針になっています。

たとえば、プロジェクト全体の構造を見るための `project_overview_view` があり、特定フェーズを詳しく見るための `phase_detail_view` があります。個別 task の編集を意識した `task_edit_view` もあります。

今回のように、新しくWBSの叩き台を作りたいときには `project_draft_view` が中心になります。生成AIがこの形式の JSON を返し、`mikuproject` がそれを取り込むことで、WBSとして扱える形に落とし込めます。

つまり、`mikuproject-ai-json-spec` は単なるデータ形式ではなく、生成AIと `mikuproject` を行き来しながら計画を作っていくための橋渡しの仕様だと捉えています。

## 生成AIと `mikuproject` を行き来してWBSを作る

今回やりたいのは、生成AIにWBSの叩き台を考えさせ、その結果を `mikuproject` で受け取り、実務で見やすい形に変換することです。

流れとしては単純です。

1. `mikuproject-ai-json-spec` を生成AIに渡す
2. 計画したい内容を自然文で入力する
3. 生成AIに `project_draft_view` を返してもらう
4. その JSON を `mikuproject` に取り込む
5. `WBS Excel ブック (.xlsx)` や `Markdown`、`SVG` などの形で出力する

ここで重要なのは、生成AIだけで完結させないことです。生成AIは叩き台を作るのが得意ですが、そのままでは実務に載せにくいことがあります。`mikuproject` を間に入れることで、WBSとして見える形へ落とし込みやすくなります。

逆に言えば、`mikuproject` 単体で最初から全部を手入力するのでもなく、生成AIと `mikuproject` を往復しながら計画を作る、というのが今回の主題です。

## 生成されたWBS関連アウトプット

`mikuproject` に取り込んだあとは、同じWBS情報をいくつかの形で扱えるようになります。

- `WBS Excel ブック (.xlsx)`
- `WBS Markdown`
- `SVG`
- 必要に応じた JSON 系の出力

このあたりは、単に見た目を変えるだけではなく、用途ごとに持ち出しやすい形へ変換できるところに意味があります。レビューしたいとき、共有したいとき、図として貼りたいとき、それぞれで向いた形式が違うからです。

今回は、生成AIで作った叩き台が、こうしたWBS関連アウトプットへ実際につながるところまでを確認します。

## まとめ

WBSは、計画を立てて実行に移すうえで重要ですが、作るのは意外と面倒です。

そして生成AIを使うことが自然になってきた今、計画策定の道具もまた、生成AIとの往復を前提にした構造へ少しずつ変わっていく余地があると感じています。

`mikuproject-ai-json-spec` と `mikuproject` は、そのためのひとつの試みです。今回の記事では、生成AIと `mikuproject` を行き来しながらWBSの叩き台を作り、それを `WBS Excel ブック (.xlsx)` や `Markdown`、`SVG` といった形へ落とし込めるところまでを見ていきます。
